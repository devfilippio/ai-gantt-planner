## 1. Задача и критерий успеха

Full-stack AI-native продукт: веб-страница с интерактивной диаграммой Гантта. Пользователь сразу видит план с сид-данными; рядом чат, где LLM-агент массово редактирует план на естественном языке (перенос задач, зависимости, добавление, переназначение исполнителей). Правки агента мгновенно отражаются на диаграмме. Клик по задаче — модалка с деталями. Импорт/экспорт Excel.

**Критерий успеха (планка владельца):** «однозначно вау». Это значит: продукт выглядит как инструмент инженера уровня Vercel/Linear (не AI-поделка), MCP — реальный архитектурный слой (а не «для галочки»), а правки агента визуально анимированы так, что демо-ролик запоминается. От качества зависит карьера владельца.

**Обязательный стек (из ТЗ):** React, Python/FastAPI, MCP, LLM через API.

## 2. Архитектура

Монорепо, один проект Vercel, один URL. Статичный React-фронт + Python-функции FastAPI под `/api`.

```
frontend/                # Vite + React 19 + TypeScript + Zustand
  src/
    components/          # GanttChart, ChatPanel, TaskModal, Toolbar
    gantt/              # чистый рендер-слой (расчёт координат, стрелки)
    store/              # Zustand: план, применение патчей, undo-состояние
    api/                # клиент REST + SSE
api/
  index.py              # FastAPI app, роуты
  scheduler.py          # топологическая сортировка, расчёт дат, критический путь, циклы
  excel.py              # import/export (openpyxl)
  agent.py              # LLM-агент: OpenRouter, цикл tool-calling, SSE-стрим
  tools.py              # ЕДИНЫЙ набор инструментов (используют и агент, и MCP)
  mcp_server.py         # MCP streamable HTTP на /api/mcp (официальный Python SDK)
  db.py                 # доступ к Neon Postgres
  models.py             # Pydantic-схемы (Task, Plan, PlanPatch, ...)
sample-data/plan.xlsx   # пример Excel для проверки
docs/
  roadmap-to-production.md
vercel.json
README.md
```

**Почему так:** один деплой, ноль DevOps, FastAPI официально поддерживается Vercel (Python 3.13, Fluid Compute). Единый `tools.py` — ключевое решение: один и тот же код инструментов вызывается и внутренним агентом, и MCP-сервером, поэтому MCP не дублирует логику, а буквально выставляет наружу то же ядро.

## 3. Модель данных

**Task:**
- `id: str` — стабильный (slug или uuid)
- `name: str`
- `description: str`
- `assignee: str`
- `duration_days: int` (> 0)
- `predecessors: list[str]` — id задач-предшественников
- `color_hint: str | None` — опционально для группировки

**Даты не хранятся** — вычисляются планировщиком:
- `start = max(end всех предшественников)` либо старт проекта, если предшественников нет
- `end = start + duration_days`

**Хранение (Neon Postgres):**
- Таблица `tasks` — текущее состояние плана.
- Таблица `plan_snapshots` — снапшот всего плана перед каждым ходом агента (для undo).

Serverless не держит состояние в памяти между вызовами, поэтому БД обязательна. Это же плюс для «Roadmap to production» — настоящая БД с первого дня.

## 4. Планировщик (`scheduler.py`)

Чистые функции, без БД и сети — легко тестируются.

- `compute_schedule(tasks) -> dict[task_id, {start, end}]` — топологическая сортировка (Kahn), затем расчёт дат.
- **Детекция циклов:** если топосорт не покрывает все задачи — есть цикл; вернуть ошибку с перечислением задач в цикле («A → B → A»).
- **Критический путь:** самый длинный путь по длительностям от старта до финиша; помечаем задачи `is_critical`. Дёшево (тот же граф), выглядит профессионально, подсвечивается на диаграмме.
- Валидация: `duration_days > 0`, все `predecessors` существуют, нет самоссылок.

## 5. Excel (`excel.py`)

Формат из ТЗ. Колонки (русские заголовки): **задача, описание, исполнитель, длительность, предшественники**. Предшественники — имена задач через запятую/точку с запятой.

- **Импорт:** читать через openpyxl, сопоставлять имена предшественников с задачами, валидировать. Ошибки — с номером строки: «строка 7: предшественник "X" не найден».
- **Экспорт:** тот же формат обратно (round-trip). Тест round-trip обязателен.
- Пример `sample-data/plan.xlsx` генерируется из сид-данных.

## 6. Инструменты агента и MCP (`tools.py`)

Единый набор, каждый инструмент — чистая функция над планом, возвращающая `PlanPatch`:

- `get_plan()` — текущий план.
- `add_task(name, description, assignee, duration_days, predecessors)`
- `update_task(id, ...поля)`
- `delete_task(id)`
- `set_dependencies(id, predecessors)`
- `reassign_tasks(filter, new_assignee)` — массово («все задачи Иванова → Петров»).
- `shift_tasks(filter, days)` — массово («перенеси задачи Олега на неделю»).
- `undo_last_turn()` — откат к снапшоту перед последним ходом агента.

После каждой мутации — пересчёт расписания (детекция циклов; при цикле мутация отклоняется с объяснением, план не портится).

**Два потребителя одного кода:**
1. **Внутренний агент** (`agent.py`, роут `POST /api/chat`): цикл LLM ↔ tools через OpenRouter. Каждый вызов инструмента стримится фронту SSE-событием.
2. **MCP-сервер** (`mcp_server.py`, роут `/api/mcp`, официальный Python MCP SDK, streamable HTTP): те же инструменты доступны Claude Desktop / Cursor. В README — инструкция подключения. Это и есть «MCP как слой».

## 7. Агент и стриминг (`agent.py`)

- **LLM:** `anthropic/claude-sonnet-4.5` через OpenRouter (лучший tool-calling), фолбэк `openai/gpt-4o` при ошибке/недоступности. Ключ — env `OPENROUTER_API_KEY`.
- **Роут `POST /api/chat`:** принимает сообщение + текущий план, запускает цикл tool-calling, отдаёт **SSE-стрим** событий:
  - `{type: "tool_call", tool, args}` — агент вызвал инструмент;
  - `{type: "patch", plan_patch}` — изменение плана (фронт применяет сразу);
  - `{type: "message", text}` — текстовый ответ агента;
  - `{type: "done"}` / `{type: "error", detail}`.
- **Undo:** снапшот плана делается перед первым мутирующим вызовом хода.
- **Guardrails (системный промпт):** агент работает только с планом через инструменты; не выполняет команд вне домена; при неоднозначности — уточняет, а не гадает; отклонённые мутации (цикл, несуществующий id) объясняет пользователю. Пишем сами (внешний guardrails-хук не используем).

**Почему SSE, а не WebSocket:** на Vercel serverless нет постоянных сокетов, а SSE закрывает требование «мгновенно». Пользователь видит, как агент правит план по шагам — это сильнее простого «применилось разом».

## 9. Обработка ошибок

- **Excel-импорт:** валидация с номерами строк; при ошибке — понятное сообщение, план не меняется.
- **Циклы зависимостей:** мутация/импорт отклоняются с перечислением цикла; в чате — объяснение.
- **LLM недоступен:** фолбэк-модель, затем честное сообщение в чат.
- **Ручные правки:** оптимистичны; при ошибке API откатываются к прошлому состоянию.
- **Пустой/битый файл:** отдельное сообщение, не 500.

## 10. Тестирование

Два уровня, оба обязательны и оба закоммичены в репозиторий:

**Уровень 1 — юнит (pytest, TDD red-green):** `scheduler` (расчёт дат, критический путь), детекция циклов, Excel round-trip, каждый инструмент из `tools.py`, реестр инструментов, хранилище (snapshot/undo), REST-эндпоинты (TestClient), цикл агента (LLM мокается — тестируем мутации плана, не модель).

**Уровень 2 — E2E (Playwright, `frontend/e2e/`):** настоящие прогоняемые спеки, а не ad-hoc скриншоты. Запускают реальный бэкенд + собранный фронт и проверяют пользовательские сценарии через браузер:
- **Золотой путь из ТЗ:** импорт `sample-data/plan.xlsx` → правка через чат → экспорт (проверяем, что задачи загрузились, бар анимированно сдвинулся после команды агента, файл экспортировался).
- Клик по задаче → модалка с корректными датами.
- Массовая команда агента → несколько баров меняются, появляется чип tool-call.
- Отмена (undo) возвращает план.
- Ошибка импорта (битый Excel) → тост с построчным сообщением.
- Адаптив: рендер на 390px и 1440px.

**Детерминизм E2E:** чат-сценарии не должны зависеть от живого LLM (флейки + расход токенов). Бэкенд поднимается с `ENV=test` / `MOCK_LLM=1`, где агент использует сценированный (canned) ответ вместо OpenRouter — тот же seam, что и в юнит-тесте агента. Отдельная «дымовая» проверка против реального OpenRouter — ручная, вне CI.

- CI (GitHub Actions) прогоняет оба уровня — вынесен в Roadmap как желательный, но в рамках теста настраивается опционально, если останется время; локальный прогон обоих уровней обязателен перед сдачей.
